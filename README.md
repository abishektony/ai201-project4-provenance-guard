# Provenance Guard

A content-attribution service that estimates whether a piece of text was written by a
human or generated by an AI model, attaches a plain-language transparency label, records
every decision in a structured audit log, and lets creators appeal a classification for
human review.

A **3-signal ensemble** feeds a single calibrated confidence score, which maps to one of
three verdicts: `likely_human`, `uncertain`, or `likely_ai`.

## Architecture Overview

The path a submission takes from input to the label a user sees:

```text
        +-------------------+
        |   POST /submit    |   raw text + creator_id
        +---------+---------+
                  |
                  v
   +--------------+--------------+
   |   Rate Limiter (10/min)     |   rejects floods with HTTP 429
   +--------------+--------------+
                  |
                  v
    +-------------+--------------------------+
    | 3-Signal Ensemble                       |
    | - Signal 1: TTR (lexical diversity)     |  -> stylometric_score [0..1]
    | - Signal 2: Groq LLM (holistic)         |  -> llm_score        [0..1]
    | - Signal 3: Burstiness (sent. variance) |  -> burstiness_score [0..1] | abstain
    +-------------+--------------------------+
                  |  weighted vote: 0.60 llm / 0.25 burstiness / 0.15 ttr
                  |  (abstaining signals drop out; weight redistributes)
                  v
   +--------------+--------------+
   | Confidence Engine           |  -> combined_score + confidence
   +--------------+--------------+
                  |  thresholds: <0.40 human | 0.40-0.60 uncertain | >0.60 ai
                  v
   +--------------+--------------+
   | Transparency Label          |  -> human-readable label text
   +--------------+--------------+
                  |
                  v
   +--------------+--------------+        +------------------+
   | In-Memory Audit Log         |<-------|  POST /appeal    |  content_id + reasoning
   +--------------+--------------+        +------------------+   sets status -> under_review
                  |
        +---------+---------+
        |                   |
        v                   v
   JSON response      GET /analytics + /dashboard  (derived metrics view)
```

**Submission flow:** A `POST /submit` request passes the rate limiter, then all three
signals run on the raw text. Their scores are combined by a weighted vote into a single
`combined_score`, bucketed into a verdict, and turned into a label; the full decision
(all signal scores, the combined confidence, the verdict, a UUID `content_id`, and a
timestamp) is written to the audit log before the JSON response returns.

**Appeal flow:** A `POST /appeal` with a known `content_id` flips that entry's status to
`under_review` and attaches the creator's reasoning, so a human reviewer can find it via
`GET /log`.

## Detection Signals

Every signal has a blind spot. Naming them honestly is part of the design.

* **Signal 1 — Type-Token Ratio heuristic (pure Python).**
  *Measures:* lexical diversity, `unique words / total words`, mapped to an AI-likelihood
  score. *Why chosen:* fast, free, deterministic, no network — a cheap sanity check.
  *Blind spot:* **weak and sometimes inverted on formal long-form text** — polished AI
  prose uses *varied* vocabulary, so TTR reads it as "human." It also misfires on
  deliberately repetitive human writing (poetry refrains), scoring it as AI.

* **Signal 2 — Groq `llama-3.3-70b-versatile` assessment.**
  *Measures:* holistic "AI-ness" — syntactic predictability, templated phrasing, tonal
  uniformity — as a 0–1 probability. *Why chosen:* captures semantic/structural patterns
  a word-count heuristic cannot, and is the only signal that reliably catches fluent,
  lexically rich AI text. *Blind spot:* highly formal human writing (legal, academic,
  ESL) shares the predictable structure of LLM output and is often rated as automated.

* **Signal 3 — Burstiness (sentence-length variance, pure Python).**
  *Measures:* how much sentence length varies (coefficient of variation of sentence
  word-counts). Human writing is "bursty" — it mixes long and short sentences; AI prose
  trends toward uniform lengths, so low variance → high AI score. *Why chosen:* it is a
  structural cue that is *orthogonal* to vocabulary and to the LLM's judgment, so it can
  break ties the other two get wrong. *Blind spot:* unreliable on very short inputs, so
  it **abstains** below 4 sentences rather than guessing.

If `GROQ_API_KEY` is missing, Signal 2 degrades gracefully to a neutral `0.5` and logs a
warning — the service keeps running on the two offline signals.

## Confidence Scoring

The signals are combined by a **weighted vote** on a scale where `0.0` = strongly human
and `1.0` = strongly AI:

```
combined_score = (0.60·llm + 0.25·burstiness + 0.15·ttr)   ÷  sum of active weights
```

**Conflict resolution / abstention.** The LLM leads (0.60) because it is by far the most
reliable signal; burstiness (0.25) outranks the weak TTR heuristic (0.15). When signals
*disagree*, the weighted average is the resolution — e.g. if the LLM says AI (0.8) but
burstiness leans human (0.34), the blend can land in `uncertain` rather than forcing a
side (see the validation table, row 4). When a signal **abstains** (burstiness on short
text), it drops out of the vote entirely and its weight is redistributed across the
remaining signals, so abstention never dilutes a confident verdict toward 0.5.

**Confidence is decoupled from the verdict.** It measures how far the score sits from a
0.5 coin-flip, scaled to [0, 1]:

```
confidence = |combined_score - 0.5| × 2
```

A score of 0.5 yields confidence **0** (maximally uncertain); 0.0 or 1.0 yields
confidence **1.0**. This is deliberate: an `uncertain` verdict always carries *low*
confidence, so the system is honest that it doesn't know rather than reporting a
misleadingly high number.

| `combined_score` | verdict | meaning |
|---|---|---|
| `> 0.60` | `likely_ai` | signals lean AI |
| `0.40 – 0.60` | `uncertain` | protected gray zone — no penalty applied to creators |
| `< 0.40` | `likely_human` | signals lean human |

### Validation — does the score vary meaningfully?

Real output from the demo battery (`python demo.py`), with individual signal scores
shown next to the ensemble result:

| Input | `llm` | `burstiness` | `ttr` | combined | verdict | confidence |
|---|---|---|---|---|---|---|
| Clearly AI (formal, templated) | 0.80 | abstain | 0.16 | 0.67 | `likely_ai` | 0.34 |
| Clearly human (casual, irregular) | 0.40 | 0.00 | 0.10 | 0.26 | `likely_human` | 0.49 |
| Borderline — formal human / academic | 0.80 | abstain | 0.19 | 0.68 | `likely_ai` | 0.36 |
| **Signal conflict** — climate advocacy | 0.80 | 0.34 | 0.14 | 0.59 | `uncertain` | 0.17 |
| Long uniform AI (burstiness fires) | 0.87 | 1.00 | 0.22 | 0.80 | `likely_ai` | 0.61 |

The clearly-AI and clearly-human inputs land in different buckets with clearly different
scores; the **conflict** row shows the ensemble doing its job — the LLM says AI but
burstiness pulls it into `uncertain`; and the **long uniform AI** row shows burstiness
firing (1.00) to raise confidence on a case the short-text inputs couldn't resolve.

### Two example submissions with different confidence

* **Higher confidence — `likely_ai` @ 0.61:** the long uniform-sentence text. All three
  signals agree (`llm` 0.87, `burstiness` 1.00, `ttr` 0.22 → combined 0.80), so the score
  sits far from 0.5 and confidence is high.
* **Lower confidence — `uncertain` @ 0.17:** the climate-advocacy paragraph. The LLM
  leans AI but burstiness disagrees, so the combined score (0.59) lands just inside the
  gray band near the 0.5 coin-flip — low confidence, correctly signalling "I'm not sure."

## Transparency Label

The label text is generated from the verdict and confidence. All three variants are
reachable (demo confidence values shown):

* **High-confidence AI (`likely_ai`):**
  `Automated Signature Identified: This entry demonstrates structural uniformity typical of AI generation (Confidence: 61%).`
* **High-confidence human (`likely_human`):**
  `Authenticated Creative Human Work: This prose presents distinctive diversity and structural variation matching human craft (Confidence: 49%).`
* **Uncertain (`uncertain`):**
  `Mixed Markers Detected: This submission exhibits blending styles. Content origin cannot be definitively verified automatically.`

The label varies by confidence in two ways: the verdict template changes entirely across
the three bands, **and** the percentage in the text reflects the confidence (e.g. a
high-confidence AI result reads "Confidence: 61%", a borderline one much lower). No
jargon — a non-technical reader sees "Automated Signature Identified," not a logit.

## Rate Limiting

Applied to `POST /submit` via Flask-Limiter (in-memory storage), keyed by client IP:

```python
@limiter.limit("10 per minute; 100 per day")
```

**Reasoning for these specific values:** a genuine creator submits their own work a
handful of times — drafting, revising, re-checking — comfortably under **10 per minute**.
That ceiling still blocks a script trying to flood the paid, latency-bound Groq endpoint.
The **100 per day** cap is a second backstop against a slow-drip abuser who stays under
the per-minute limit: 100 checks/day is generous for one author but caps daily cost and
the blast radius of a leaked client. Both are per-IP so one bad actor can't starve
everyone else.

**Evidence** (12 rapid requests against a fresh server; limit is 10/min):

```text
request 1  -> 200      request 7  -> 200
request 2  -> 200      request 8  -> 200
request 3  -> 200      request 9  -> 200
request 4  -> 200      request 10 -> 200
request 5  -> 200      request 11 -> 429
request 6  -> 200      request 12 -> 429
```

The first 10 succeed; everything past the limit returns HTTP `429 Too Many Requests`.

## Appeals Workflow

Any creator whose submission exists in the audit log can appeal by sending its
`content_id` plus their reasoning to `POST /appeal`. The system flips that entry's
`status` to `under_review` and stores the `appeal_reasoning` alongside the original
classification. A human reviewer opens the queue via `GET /log` and sees every
`under_review` item with its original signals, score, and the creator's argument.
(Automated re-classification is intentionally out of scope.)

## Stretch Features

### ✅ Ensemble Detection (+1)
The pipeline uses **three** distinct signals (TTR, Groq LLM, burstiness) combined by a
documented **weighted vote** (`0.60 / 0.25 / 0.15`). Conflict resolution is explicit: the
weighted average decides, the LLM is given the most authority, and a signal that can't
judge (burstiness on short text) **abstains** so its weight redistributes instead of
diluting the vote. The demo and audit log show every individual signal score next to the
ensemble result — see the validation table's "signal conflict" row for a case where the
signals disagree and the ensemble lands on `uncertain`.

### ✅ Analytics Dashboard (+1)
Two endpoints derive metrics from the audit log (no extra storage):
* `GET /analytics` — JSON with **4 metrics**: detection patterns (count + % per verdict),
  appeal rate, average confidence, and average per-signal scores.
* `GET /dashboard` — a simple HTML view of the same, with verdict bars and a signal table.

Sample `GET /analytics` output:
```json
{
  "total_submissions": 10,
  "detection_patterns": {
    "likely_ai":    { "count": 6, "percent": 60.0 },
    "likely_human": { "count": 4, "percent": 40.0 },

  "appeals_filed": 1,
```

## Known Limitations

* **Formal / non-native-English human prose is the most likely misclassification.** ESL
  and academic writers use regular, precise phrasing that matches an LLM's templates.
  Concretely: in the demo, a formal human paragraph on monetary policy scored `llm` 0.80
  and was flagged `likely_ai`. This is a property of *both* the LLM and TTR signals (the
  LLM keys on predictable structure; TTR is near-useless here), not a data-volume problem,
  and it is exactly the case the appeals path exists to catch.
* **Deliberately repetitive human writing** (poetry refrains, minimalist micro-fiction)
  has low lexical diversity, so Signal 1 spikes toward AI even for genuine human work.
* **Short inputs** lose the burstiness signal entirely (it abstains under 4 sentences), so
  very short submissions are judged on the LLM and TTR alone.

## Spec Reflection

* **How the spec helped:** writing the three label variants and the threshold table in
  `planning.md` *before* coding gave the confidence engine a concrete target — fixed
  `<0.40 / 0.40–0.60 / >0.60` bands and exact label strings — so there was no ambiguity
  mid-build and the three-way (non-binary) verdict fell out naturally.
* **How the implementation diverged, and why:** my spec fixed the blend at `0.40 / 0.60`
  (TTR / LLM). Real testing proved that wrong — TTR is weak and sometimes inverted on
  formal text, and at 40% weight it pulled a correctly AI-flagged sample (`llm` 0.80) into
  `uncertain`. I re-weighted, then went further and added a **third signal (burstiness)**
  as a stretch feature, landing on a `0.60 / 0.25 / 0.15` ensemble with abstention. The
  spec's *thresholds and labels* survived intact; only the empirically chosen weighting
  scheme changed — the kind of calibration you can only validate with real inputs.

## AI Usage

* **Scaffolding the pipeline (M3–M4).** I directed the AI to generate the Flask skeleton
  with the `/submit` route plus the signal functions from my spec and diagram. It produced
  a working handler, but I **overrode the confidence math**: its first version flipped at
  0.5 and set `confidence = combined_score`, which gave an `uncertain` result a *high*
  confidence. I replaced it with the decoupled `|score − 0.5| × 2` formula.
* **Debugging the dead second signal.** Every submission returned `llm_score: 0.5`. I had
  the AI find why; it traced the cause to a silent `except: return 0.5` swallowing a Groq
  `401 Invalid API Key`. I **revised the error handling** to log the real exception (plus a
  startup warning when the key is absent) — which is what surfaced that my key was invalid.
* **Ensemble design decision (overrode the AI's first cut).** When I added burstiness, the
  AI's initial version made it vote `0.5` when it couldn't judge short text. That *diluted*
  confident verdicts and dropped a clear AI sample to `uncertain`. I **changed it to
  abstain** and redistribute its weight across the other signals instead — a correctness
  fix the naive weighted-average had hidden.

## Setup & Run

Requires Python 3.13+. This repo uses [uv](https://docs.astral.sh/uv/); plain `pip` works.

```bash
# 1. Install dependencies
uv sync                      # or: pip install flask flask-limiter groq python-dotenv

# 2. Configure the Groq API key  (free key at https://console.groq.com/keys)
cp .env.example .env         # then edit .env: GROQ_API_KEY=gsk_...

# 3a. Run the server
uv run python app.py         # serves on http://localhost:5000

# 3b. OR run the full end-to-end demo (starts the server itself)
uv run python demo.py
```

`demo.py` exercises every feature — all five scoring scenarios with per-signal scores, an
appeal, the rate limiter, the audit log, and the analytics dashboard — and is the script
used to drive the walkthrough video.

## API Reference

### `POST /submit`
Classify text. Rate-limited to 10/min, 100/day per IP.
Request: `{ "text": "...", "creator_id": "user-123" }`
Response `200`: `{ content_id, attribution, confidence, transparency_label }`.
`400` if `text`/`creator_id` missing, `429` if rate-limited.

### `POST /appeal`
Contest a classification. Request: `{ "content_id": "...", "creator_reasoning": "..." }`.
`200` confirms status is now `under_review`. `400` for missing fields, `404` for unknown id.

### `GET /log`
All audit-log entries as `{ "entries": [...] }`.

### `GET /analytics`
Derived metrics as JSON (detection patterns, appeal rate, average confidence, average
signal scores).

### `GET /dashboard`
HTML view of the analytics metrics for a browser.

> `GET /log`, `/analytics`, and `/dashboard` are open here for grading/documentation
> visibility; a real deployment would put them behind auth.
